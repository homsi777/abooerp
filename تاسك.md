## 1. Current Confirmed Business Rules

The project must preserve these rules:

### Shipments

- Freight charge is company shipping income.
- COD / collection amount is money collected for the sender/customer.
- Prepaid amount means money was already received by the company.
- Transfer service fee is separate from freight, COD, prepaid, and additional charges.
- Daily Shipping Book must keep financial fields clear and separated.

### Agents

- Agent commission is calculated only from shipment freight charge.
- Agent commission applies whether freight is collected on delivery or prepaid by sender.
- Agent commission must not be calculated from:
  - COD
  - transfer service fee
  - transfer amount
  - total shipment amount
  - original_amount
  - additional charges
  - company transfer profit
- Agent commission must be stored historically using snapshot fields.
- Changing agent commission percentage later must not affect old shipments.

### Transfers / Hawala

- Transfer service fee belongs only to the main branch / company headquarters.
- Transfer service fee is company/headquarters profit.
- Transfer service fee must not be agent income.
- Transfer service fee must not affect agent commission.
- A transfer can be standalone or linked to a shipment.
- Shipment-linked transfer service fee from Daily Shipping Book must create a linked PENDING transfer.
- Transfer posting happens only when transfer is completed.
- Completing transfer must create confirmed receipt voucher and cashbox transaction.
- Cancelling completed transfer must create reversal entries.

## 2. Critical Fix 1 — Remove or Lock Down x-user-id Auth Bypass

The final readiness report identified a critical security issue:

x-user-id can build user context without proper JWT/session validation.

This must be fixed before real deployment.

Inspect:

- server/src/middleware/requestContext.ts
- authentication/session/JWT middleware
- all places where request user context is built
- any frontend/backend code using x-user-id

Required behavior:

1. In production, backend must never trust x-user-id from client requests.
2. User identity must come from authenticated session/JWT/secure auth only.
3. If x-user-id is needed for local development, allow it only when all conditions are true:
   - NODE_ENV !== 'production'
   - request is from localhost only
   - explicit env flag is enabled, such as ALLOW_DEV_USER_HEADER=true
4. In production/LAN deployment:
   - ignore x-user-id
   - or reject the request if it tries to use it without valid auth
5. Add clear code comments explaining this is development-only if kept.
6. Ensure all protected endpoints still work with the normal authentication flow.
7. Do not break login.
8. Do not break role/permission checks.
9. Do not break agent scoping.

Validation:

- Valid authenticated request works.
- Request without authentication fails.
- Request with only x-user-id fails in production mode.
- Request with x-user-id works only in explicitly allowed local development mode.
- Permissions still work correctly after the change.

## 3. Critical Fix 2 — Make Transfer Delete Accounting-Safe

The final report identified a high risk:

Transfers can still be hard-deleted when not completed.

This is not ideal for accounting traceability.

Inspect:

- server/src/routes/transfers.ts
- server/src/services/transfersService.ts
- server/src/repositories/transfersRepository.ts
- transfer migrations
- audit logging service
- Transfers UI delete/cancel buttons

Required behavior:

1. Completed transfers must never be hard-deleted.
2. Cancelled transfers must never be hard-deleted.
3. Pending transfers should preferably be cancelled, not hard-deleted.
4. Replace destructive delete UX with “Cancel Transfer” where possible.
5. If hard delete is still kept for very limited cleanup cases, it must:
   - be blocked for completed transfers
   - be blocked for cancelled transfers
   - require transfers.delete
   - create audit log
   - be documented as non-accounting cleanup only
6. Preferred implementation:
   - convert delete action into cancellation for PENDING transfers
   - use status CANCELLED
   - store cancellation reason if available
   - store cancelled_by/cancelled_at if schema supports it
7. Completed transfer cancellation must use existing V2 reversal behavior.
8. Transfers UI must not show dangerous delete button for completed/cancelled transfers.
9. Transfers UI should clearly show:
   - Pending
   - Completed
   - Cancelled
   - receipt voucher number
   - cashbox name
   - cancellation reason if available

Validation:

- Pending transfer can be cancelled safely.
- Completed transfer cannot be hard-deleted.
- Completed transfer cancellation triggers existing reversal flow.
- Cancelled transfer remains visible in list/report.
- Audit log records cancellation/delete action.
- Transfers list still works.
- Transfer complete/cancel posting remains intact.

## 4. Critical Fix 3 — Enforce Agent Commission From Freight Charge Only

The final report found a medium accounting risk:

Agent commission calculation may fallback to original_amount if freight_charge is missing.

This must be corrected.

Business rule:

Agent commission must be calculated only from freight_charge.

Never calculate agent commission from:

- original_amount
- total shipment amount
- COD
- transfer service fee
- transfer amount
- additional charges
- company transfer profit

Inspect:

- server/src/services/shipmentService.ts
- server/src/repositories/shipmentRepository.ts
- shipment create/update logic
- Daily Shipping Book posting logic
- agent commission snapshot logic
- Agent COD Statement logic

Required behavior:

1. Remove unsafe fallback from original_amount for agent commission base.
2. If a shipment has an assigned agent and commission should be calculated:
   - use freight_charge only.
3. If freight_charge is missing:
   - either block creation/update with clear validation error
   - or explicitly set commission to 0 with clear reason
   - choose the safest option that does not break existing valid workflows.
4. If freight_charge = 0, commission must be 0 unless business rules later say otherwise.
5. Snapshot fields must remain:
   - agent_commission_base_type = FREIGHT_CHARGE
   - agent_commission_base_amount = freight_charge
   - agent_commission_percentage_snapshot
   - agent_commission_amount_snapshot
6. Changing agent commission percentage later must not affect old shipments.
7. Transfer service fee must not affect commission.
8. COD must not affect commission.
9. Additional charges must not affect commission.

Validation scenarios:

Scenario 1:

freight_charge = 100
agent_commission_percentage = 10
expected agent commission = 10

Scenario 2:

freight_charge = 0
original_amount = 500
agent_commission_percentage = 10
expected agent commission must NOT become 50

Scenario 3:

freight_charge missing
original_amount = 500
agent_commission_percentage = 10
expected: block with clear error OR commission = 0 with explicit safe reason

Scenario 4:

freight_charge = 100
transfer_service_fee = 20
agent_commission_percentage = 10
expected agent commission = 10
transfer service fee must not affect commission
